# Slice s2 — Minimal Dockerfile + .dockerignore for the dashboard server

## Goal (this slice only)

Add a minimal `Dockerfile` and `.dockerignore` at the repo root that bake in the
EXISTING `scripts/dashboard_server.py` + `scripts/dashboard_assets/` and run the
existing server inside the container. No compose, no multi-arch, no healthcheck, no
registry publish. Stdlib only (no `pip install`). Non-root user. Assets baked in
(never mounted). Default CMD runs the existing server with `--bind-host 0.0.0.0`
and a sensible default `--port`, and does NOT pass `--advertise-port` (s3 supplies
it at runtime) nor widen the Host allowlist.

## Context (verified against the base at alpha @ cd58f06)

- `scripts/dashboard_server.py` `main()` accepts: `--port` (default `8787` = `DEFAULT_PORT`),
  `--root` (append, default cwd), `--bind-host` (default `127.0.0.1`),
  `--advertise-port` (default `None`, drives Host allowlist).
- `build_server` auto-resolves assets from `Path(__file__).resolve().parent / "dashboard_assets"`,
  so co-locating the script and `dashboard_assets/` in the image is sufficient — no
  `--assets` flag needed. Assets = `scripts/dashboard_assets/index.html`.
- Server is pure stdlib (`http.server`, `argparse`, etc.). Python 3.14 locally; CI uses 3.12.
- CI (`.github/workflows/validate.yml`) has NO Docker daemon — must not add a
  Docker-build/run CI test.
- `.gitignore` already ignores `.worktrees/`, `__pycache__/`, `*.pyc`.

## Design decisions

- Base image: `python:3.12-slim` (matches CI's pinned 3.12; slim keeps it lean;
  stdlib only so no build toolchain needed). Pin a digest is out of scope for a
  minimal image; pin the tag to a specific minor (`3.12-slim`).
- Non-root: create a dedicated unprivileged user (`app`, uid/gid 10001) and `USER` it
  before CMD. Do NOT run as root.
- Copy ONLY runtime files: `scripts/dashboard_server.py` and
  `scripts/dashboard_assets/` into the image under `/app/scripts/` so the server's
  `__file__`-relative asset resolution works unchanged. WORKDIR `/app`.
- CMD: `["python3", "scripts/dashboard_server.py", "--bind-host", "0.0.0.0", "--port", "8787"]`.
  No `--advertise-port` (s3 adds it), no extra `--root` (defaults to WORKDIR `/app`;
  s3 bind-mounts artifact roots `:ro` at runtime). `EXPOSE 8787` for documentation.
- Header comment documenting the REQUIRED run contract s3 must enforce:
  `-p 127.0.0.1:PORT:PORT` (loopback only), artifacts mounted `:ro`, `--cap-drop ALL`,
  non-root (already baked), no `--privileged`, no docker.sock mount.
- `.dockerignore`: default-exclude everything (`*`), then re-include ONLY the two
  runtime paths (`!scripts/dashboard_server.py`, `!scripts/dashboard_assets/` +
  its subtree) — NOT a blanket `!scripts/` (which would readmit tests,
  `pr_resolver.py`, `release.py`, `validate_marketplace.py`, `__pycache__`). This
  guarantees `.git`, `.worktrees`, `docs/`, tests, `__pycache__`, and run artifacts
  never enter the build context or image. [FOLDED: skeptic/architect/guardian]
- Header comment also documents that with the default CMD (no `--root`), cwd
  defaults to `/app` and `/app/docs/spec-loop` does NOT exist → the server starts
  healthy (stderr warning) and serves zero runs until s3 bind-mounts artifact roots
  `:ro` and passes `--root <mounted-path>`. So an empty dashboard is not a bug.
  [FOLDED: skeptic/architect/guardian]

## Testing strategy (no Docker in CI)

- No CI test that builds/runs the image (no daemon in CI; hard constraint).
- Static verification only, consistent with the repo's stdlib-unittest style, is
  OPTIONAL for this 2-file infra slice; given Tier-2 + review coverage, rely on
  review + a local `docker build` as report evidence rather than adding a bespoke
  static test file (which would be net-new surface outside the two named files).
  Files are constrained to `Dockerfile` and `.dockerignore` only.
- Local evidence: run `docker build` locally (if a daemon is available) to prove the
  image builds and `python3 scripts/dashboard_server.py --help` resolves inside it.
  This must NOT become a CI dependency.

## Steps (bite-sized)

1. Write `.dockerignore` (deny-all, allow only `scripts/`) — ensures a lean context.
2. Write `Dockerfile`: `python:3.12-slim` base, WORKDIR `/app`, non-root user
   (uid/gid 10001), COPY `scripts/dashboard_server.py` + `scripts/dashboard_assets/`
   into `/app/scripts/`, `EXPOSE 8787`, run-contract header comment, `USER app`,
   `CMD` as above.
3. Local verification (best-effort): `docker build -t spec-loop-dashboard:s2 .`,
   then `docker run --rm spec-loop-dashboard:s2 python3 scripts/dashboard_server.py --help`
   to confirm the baked server + assets resolve and it runs non-root. Record output
   for the report. If no daemon, note "no local daemon; relied on review".
4. Verify baseline Python tests still pass (unchanged; sanity only):
   `python3 scripts/test_dashboard_server.py`.
