## Root-cause finding that shapes this plan (measured, not assumed)

`scripts/measure_coverage.py` runs the suite under `trace.Trace`, which **traces only the calling thread**. The existing HTTP tests exercise the handler over a real socket via `serve_forever` on a **daemon thread**, so the entire request/response code path (`do_GET`, `_route`, `_serve_static`, `_read_asset`, `_emit`, `_content_type`, `_json_response`, `_not_modified`) executes on a thread the tracer never sees and reads as UNCOVERED — even though it is behaviorally exercised. Confirmed empirically: `_emit` (515-523), `_content_type` (544), and the whole handler body show as missed lines.

**Therefore** the coverage-raising lever is to exercise the handler methods and helper functions **synchronously on the test thread** (direct in-thread unit tests against a handler instance / the pure functions), in ADDITION to keeping the real-socket behavioral tests untouched. The real-socket tests remain the source of truth for the security invariants; the new in-thread tests make the already-correct code visible to the line-coverage tracer and add the `main()` path.

The many "missed" import-time lines (module docstring, imports, `def`/`class` headers, module constants) are counted in the denominator but can never appear in the numerator (import happens before tracing per the tool's own docstring). Those are not addressable and are not targeted.
