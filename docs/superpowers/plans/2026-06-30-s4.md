# s4 — Multi-root grouping in the dashboard overview Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Update `scripts/dashboard_assets/index.html` so the aggregated multi-root overview groups/labels each run by its source repo/root and renders s1's namespaced run-ids without collision, degrading gracefully to today's exact single-root layout when only one root is present.

**Architecture:** The overview render is the only surface that changes. A pure helper `groupRunsByRoot(runs)` partitions the API's `runs` array into ordered groups keyed by each run's `root` field. When zero runs carry a `root` field (the single-root case, where s1 guarantees byte-for-byte-identical objects: bare `run_id`, no `root`), the overview renders exactly as today — a flat list of cards, no group chrome. When any run carries a `root`, the overview renders one group section per root (a labeled `<h3>` header reusing the existing `.section`/`h3` styling) with that root's cards beneath. Cards, links, drill-down, freshness, poll loop, routing, and the detail view are untouched: the namespaced id already arrives verbatim in `run.run_id` and the existing card/link code already uses it verbatim and URL-encodes it in `navigate`/`activeUrl`, so namespaced ids flow through unchanged.

**Group order / label (settled during Iron Council):** The server emits roots in **input-argument order** (not sorted) — `_resolve_roots`/`_root_keys` preserve input order and `scan_all_roots` emits all of root-A's runs, then root-B's. The helper groups by **first-appearance order**, which matches this exactly and is robust regardless. The `<h3>` label is the **raw `root` key verbatim** — a human-meaningful, collision-free repo basename (disambiguated `#2` on collision by s1); this is the intended human affordance for "group/label by root" and is deliberately not embellished (no path hint). Confirm it reads as a group label in Step 6.

**Tech Stack:** Vanilla ES (strict mode), no framework, no external asset, self-contained single HTML file. Existing DOM helpers (`el`, `pill`, `overviewCard`, `countPills`). Node v22 (test-only, out-of-tree) for the pure-helper unit test; live server (`python3 scripts/dashboard_server.py`) for integration verification.

## Global Constraints

- READ-ONLY display only. The page issues GET only, against `/api/runs` and `/api/runs/<id>`. Do NOT add any new fetch verb or any control that could mutate/trigger a run.
- No framework, no CDN, no external asset — self-contained single file.
- Match the existing dark-theme UI voice and the 2.5s poll model (`POLL_MS`). Reuse existing patterns/helpers.
- Single-root (zero `root` fields present) MUST look/behave byte-for-byte as today: flat cards, no empty "root" chrome, no regression.

- [ ] **Step 1: Add the pure helper** near `countPills` (before `renderOverview`), matching existing style:

```javascript
    // Partition runs into ordered groups by source root. The API omits `root`
    // entirely for single-root (byte-for-byte back-compat) — that case yields
    // one empty-keyed group so the overview renders exactly as before. Any run
    // carrying a non-empty `root` flips multiRoot on and preserves first-
    // appearance root order (the server emits roots in input-argument order).
    function groupRunsByRoot(runs) {
      const groups = [];
      const byKey = Object.create(null);
      (runs || []).forEach((r) => {
        const key = (r && typeof r.root === "string") ? r.root : "";
        let g = byKey[key];
        if (!g) { g = { root: key, runs: [] }; byKey[key] = g; groups.push(g); }
        g.runs.push(r);
      });
      const multiRoot = groups.some((g) => g.root !== "");
      return { multiRoot, groups: groups.length ? groups : [{ root: "", runs: [] }] };
    }
```

- [ ] **Step 2: Branch `renderOverview` to render groups (multi-root) or flat list (single-root, unchanged)**

Replace the `runs.forEach(...)` tail of `renderOverview` so the single-root path is byte-for-byte the old behavior and multi-root adds a labeled section per root reusing `.section`/`h3`:

```javascript
    function renderOverview(runs) {
      $view.className = "";
      $view.textContent = "";
      if (!runs.length) {
        $view.className = "empty";
        $view.textContent = "no spec-loop runs found";
        return;
      }
      const { multiRoot, groups } = groupRunsByRoot(runs);
      if (!multiRoot) {
        // single root: identical to prior behavior — flat list of cards
        groups[0].runs.forEach((r) => $view.appendChild(overviewCard(r)));
        return;
      }
      groups.forEach((g) => $view.appendChild(rootGroupSection(g)));
    }
    function rootGroupSection(group) {
      const sec = el("div", { cls: "section" });
      sec.appendChild(el("h3", { text: group.root }));   // textContent — root is untrusted
      group.runs.forEach((r) => sec.appendChild(overviewCard(r)));
      return sec;
    }
```
