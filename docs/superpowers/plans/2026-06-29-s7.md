# Plan: s7 — Harden local marketplace gate to catch the YAML-frontmatter-parse-error class

## Problem
`scripts/validate_marketplace.py`'s `read_frontmatter()` is a regex KEY-PRESENCE
parser. It records `key: value` and strips the value, but never asks whether the
value is something real YAML would reject. A real bug shipped because two files had
UNQUOTED `description` scalars containing `': '` (colon-space) — which YAML parses
as a nested mapping and rejects in scalar position. They passed the local gate but
failed CI's authoritative `claude plugin validate`.

## Goal (this slice only)
Extend frontmatter validation, stdlib-only, so that for each top-level `key: value`
line, an UNQUOTED value (not wrapped in matching single/double quotes) that contains
the YAML-hostile construct `': '` is reported with a clear, file+key-named error.
Cheaply also flag, when clean: an unquoted value containing a literal tab, or an
unquoted value starting with a YAML-significant indicator char. Keep ALL existing
checks intact. This is a targeted heuristic for the known breakage class, NOT a YAML
parser — documented in a comment; CI's `claude plugin validate` remains authoritative.

## Constraints
- Only touch the two slice files. No plugin.json/CHANGELOG/README/marketplace.json.
- Stdlib only. No new deps. Do NOT shell out to the claude CLI from the script.
- Must NOT false-positive on: properly-quoted values (even containing `': '`);
  values with a colon but NO space (URLs like `https://x`); the `MultiEdit`-style
  exact-match cases already covered.

## Council folding (ENDORSE_WITH_CONCERNS 5/5 — concerns folded in)
- ARCHITECT/SKEPTIC/GUARDIAN: the existing regex `^([A-Za-z0-9_-]+):\s*(.*)$` already
  consumes LEADING whitespace via `\s*`; raw vs stripped differs only on the TRAILING
  edge. A trailing space or `\r` (CRLF-checked-out file) after a closing quote would
  break a naive `endswith(quote)` and FALSELY flag a correctly-quoted value. => Operate
  on the TRAILING-stripped value (the already-stored `m.group(2).strip()`). Drop the
  "RAW before strip" language entirely.
- ARCHITECT: original rule 3 (literal tab) is mis-justified — after strip, any surviving
  tab is INTERIOR scalar content, which is legal YAML, so flagging it diverges from real
  YAML (the exact false-positive class the constraints forbid). => DROP rule 3.
- PRAGMATIST/SKEPTIC/GUARDIAN: original rule 4 (leading indicator char `@`/`` ` ``/`%`)
  reaches past the verified breakage class; it never broke, the authoritative
  `claude plugin validate` already covers it, and a hand-picked 3-of-~18 indicator subset
  invites drift/false-positives (5 real command files have `allowed-tools` starting `[`).
  => DROP rule 4. Ship the lean core (rules 1+2) — the complete fix for the CI failure.
- HISTORIAN: extend the existing fixture builders (`cmd(quote=...)`, generalize
  `write_plugin` to optionally lay down agent/skill) rather than fork a parallel pattern.
- GUARDIAN: rule 1 MUST run before rule 2 and accept BOTH matched `"..."` and `'...'`.

## Approach (smallest change that holds the invariant)
The natural seam is `read_frontmatter()`, which already iterates the block's
top-level `key: value` lines via the existing regex and stores `m.group(2).strip()`.
Add a focused pure `@staticmethod _frontmatter_value_error(key, value)` (matching the
existing `_allowed_tools` precedent) that, given the **stripped** value, returns an
error string or None. `read_frontmatter` calls it per line on the stripped value and
records any error via `self.err`. Detection rules (lean core only):
1. If the value is wrapped in matching single OR double quotes -> SAFE (return None),
   even if it contains `': '`. (Quoting is exactly the fix YAML wants.) Evaluated FIRST.
2. Else if the value contains `': '` (colon followed by space) -> error:
   `frontmatter value for '<key>' contains ': ' and must be quoted`.
Empty values are left to the existing `require_keys` presence check — no new error.
Document in-code that this is a targeted heuristic, NOT a YAML parser, and that CI's
`claude plugin validate` remains the authoritative frontmatter gate (the known
under-detection gaps, e.g. `"a" extra` quoted-then-trailing-content, are ceded to CI).

   (an `allowed-tools` flow sequence) is NOT flagged (no rule 4).
